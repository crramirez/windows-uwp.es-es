# <a name="app-capability-declarations"></a>Declaraciones de funcionalidades de las aplicaciones

Las funcionalidades deben declararse en el [manifiesto del paquete](https://docs.microsoft.com/uwp/schemas/appxpackage/appx-package-manifest) de la aplicación de la Plataforma universal de Windows (UWP) para acceder a determinadas API o ciertos recursos como imágenes, música o dispositivos, como la cámara o el micrófono.

Para solicitar acceso a los recursos o API específicos, declara funcionalidades en el [manifiesto del paquete](https://docs.microsoft.com/uwp/schemas/appxpackage/appx-package-manifest) de tu aplicación. Puedes declarar funcionalidades generales mediante el [Diseñador de manifiestos](packaging-uwp-apps.md#configure-an-app-package) en Microsoft Visual Studio o puedes agregarlas manualmente. Para obtener más información, consulta [Cómo especificar funcionalidades en un manifiesto del paquete](https://docs.microsoft.com/uwp/schemas/appxpackage/how-to-specify-capabilities-in-a-package-manifest). Es importante saber que cuando los clientes compren tu aplicación en la Tienda, se les notificarán todas las funcionalidades que la aplicación declara. Evita declarar funcionalidades que la aplicación no necesite.

Algunas funcionalidades otorgan a las aplicaciones acceso a un *recurso con información confidencial*. Estos recursos se consideran confidenciales porque pueden acceder a datos personales del usuario o costarle dinero al usuario. La configuración de privacidad, administrada por la aplicación Configuración, permite al usuario controlar dinámicamente el acceso a recursos con información confidencial. Por lo tanto, es importante que la aplicación no suponga que un recurso con información confidencial está siempre disponible. Para obtener más información sobre cómo acceder a recursos confidenciales, consulta las [Directrices para aplicaciones compatibles con la privacidad](https://docs.microsoft.com/windows/uwp/security/index). Las capacidades que ofrecen las aplicaciones con acceso a un *resource confidencial* se anotan con un asterisco (\*) junto a la situación de capacidad.

Hay varios tipos de capacidades.

- [Funcionalidades de uso general](#general-use-capabilities), que se aplican a los escenarios de aplicación más comunes.
- [Capacidades de dispositivo](#device-capabilities), que permiten la aplicación para acceder a los dispositivos periféricos e internos.
- [Capacidades restringidas](#restricted-capabilities), que requieren aprobación para la presentación de Microsoft Store o por lo general solo están disponibles para determinados asociados y Microsoft.
- [Capacidades personalizadas](#custom-capabilities).

## <a name="device-capabilities"></a>Funcionalidades del dispositivo

Las funcionalidades de dispositivo permiten a tu aplicación acceder a los periféricos y a los dispositivos internos. Las funcionalidades de dispositivo se especifican mediante el elemento **DeviceCapability** en el manifiesto del paquete de la aplicación. Este elemento puede necesitar elementos secundarios adicionales y algunas funcionalidades de dispositivos deben agregarse al manifiesto del paquete manualmente. Para más información, consulta [Cómo especificar funcionalidades de dispositivos en un manifiesto del paquete](https://docs.microsoft.com/uwp/schemas/appxpackage/how-to-specify-device-capabilities-in-a-package-manifest) y la [**referencia de esquema DeviceCapability**](https://docs.microsoft.com/uwp/schemas/appxpackage/uapmanifestschema/element-devicecapability).

> [!NOTE]
> Puede tener varios **DeviceCapability** y **capacidad** elementos bajo la **capacidades** elemento, pero todos **DeviceCapability**elementos deben estar después de la **capacidad** elementos.

| Escenario de funcionalidad | Uso de la funcionalidad |
|---------------------|------------------|
| **Ubicación**\* | La funcionalidad **location** proporciona acceso a las funciones de ubicación, la cual se obtiene desde hardware dedicado, como un sensor GPS en el equipo, o deriva de la información de red disponible. Las aplicaciones deben administrar el caso en el que el usuario haya deshabilitado los servicios de ubicación desde el acceso a **Configuración**. |
| **Micrófono** | La funcionalidad **microphone** proporciona acceso a la alimentación de audio del micrófono, que permite a la aplicación grabar desde micrófonos conectados. Las aplicaciones deben administrar el caso en el que el usuario haya deshabilitado el micrófono desde el acceso a **Configuración**. |
| **Proximidad** | La funcionalidad **proximity** permite la comunicación entre varios dispositivos que se encuentran cerca unos de otros. Esta funcionalidad se utiliza normalmente en juegos esporádicos de varios jugadores y en aplicaciones que intercambian información. Los dispositivos intentan usar la tecnología de comunicación que proporcione la mejor conexión posible, lo que incluye Bluetooth, Wi-Fi e Internet. Esta funcionalidad se usa solo para iniciar la comunicación entre los dispositivos. |
| **Webcam** | La funcionalidad **webcam** proporciona acceso a la fuente de vídeo de una cámara integrada o a una cámara web externa, que permite a la aplicación capturar fotos y vídeos. En Windows, las aplicaciones deben controlar el caso en el que el usuario haya deshabilitado la cámara desde el acceso a **Configuración**.<br/>La funcionalidad **webcam** solo concede acceso a la secuencia de vídeo. Para conceder acceso también a la secuencia de audio, debe agregarse la funcionalidad **microphone**. |
| **USB** | La funcionalidad del dispositivo **usb** permite acceder a las API que se indican en [How to add USB device capabilities to the app manifest](https://go.microsoft.com/fwlink/p/?LinkId=302259) (Cómo agregar capacidades de dispositivo USB al manifiesto de la aplicación). |
| **Dispositivo de interfaz humana (HID)** | La funcionalidad del dispositivo **humaninterfacedevice** permite acceder a las API que se indican en [Cómo especificar funcionalidades de dispositivo para HID](https://docs.microsoft.com/uwp/schemas/appxpackage/how-to-specify-device-capabilities-for-hid). |
| **Punto de servicio (POS)** | La funcionalidad del dispositivo **pointOfService** permite acceder a las API del espacio de nombres [**Windows.Devices.PointOfService**](https://docs.microsoft.com/uwp/api/Windows.Devices.PointOfService). ## <a name="restricted-capabilities"></a>Funcionalidades restringidas

Si la aplicación declara ninguna funcionalidad restringida, debe proporcionar información durante la [proceso de envío](../publish/app-submissions.md) con el fin de ser aprobadas para publicar la aplicación en la Microsoft Store. Debe proporcionar esta información en el [opciones de envío](../publish/manage-submission-options.md#restricted-capabilities) restringido de página de su envío, que explica cómo su aplicación usa cada una funcionalidad que declara.

> [!IMPORTANT]
> Restringido capacidades están pensadas para escenarios muy concretos. El uso de estas funcionalidades se encuentra muy restringido y está sujeto a revisión y a directivas de incorporación adicionales de la Tienda. Tenga en cuenta que puede agregar o quitar aplicaciones que declaran capacidades restringidas sin necesidad de recibir cualquier aprobación. La aprobación solo es necesaria cuando se envían estas aplicaciones a la Store.

Asegúrese de no declarar que estas capacidades restricción a menos que la aplicación realmente necesita. Hay casos en los que estas funcionalidades son necesarias y apropiadas, como en aplicaciones de banca con autenticación de dos factores, en las que los usuarios proporcionan una tarjeta inteligente con un certificado digital que confirma su identidad. Es posible que otras aplicaciones se hayan diseñado principalmente para empresas y precisen acceder a recursos corporativos a los que no es posible acceder sin las credenciales de dominio del usuario.

Para declarar una funcionalidad restringida, modifique su [manifiesto del paquete de aplicación](https://docs.microsoft.com/uwp/schemas/appxpackage/appx-package-manifest) archivo de código fuente (`Package.appxmanifest`). Agregar el **xmlns:rescap** declaración de espacio de nombres XML y usar el **rescap** prefijo cuando se declara la capacidad restringida. Por ejemplo, aquí se muestra la manera de declarar la funcionalidad **appCaptureSettings**.

```xml
<?xml version="1.0" encoding="utf-8"?>
<Package
    ...
    xmlns:rescap="http://schemas.microsoft.com/appx/manifest/foundation/windows10/restrictedcapabilities"
    IgnorableNamespaces="... rescap">
...
<Capabilities>
    <rescap:Capability Name="appCaptureSettings"/>
</Capabilities>
</Package>
```

### <a name="restricted-capability-approval-process"></a>Proceso de aprobación de la funcionalidad restringida

Anteriormente, solicitábamos que te pusieras en contacto con soporte técnico para obtener la aprobación para usar una funcionalidad. Ahora permitimos que proporcione esta información en [centro de partners](https://partner.microsoft.com/dashboard/) como parte de la [proceso de envío](../publish/app-submissions.md).

Al cargar los paquetes para su envío, se detectará si se declara ninguna funcionalidad restringida. Si lo hacemos,se te pedirá que proporciones detalles acerca de cómo usa tu producto cada funcionalidad la página [Opciones de envío](../publish/manage-submission-options.md#restricted-capabilities). Asegúrate de incluir tantos detalles como sea posible para ayudarnos a comprender por qué el producto tiene que declarar la funcionalidad. Ten en cuenta que esto demora el envío para completar el proceso de certificación.

Durante el proceso de certificación, nuestros evaluadores revisarán la información que proporciones para determinar si se aprueba el envío para usar la funcionalidad. Ten en cuenta que esto demora el envío para completar el proceso de certificación. Si aprobamos el uso de la funcionalidad, tu aplicación continuará el resto del proceso de certificación. Por lo general, no tendrás que repetir el proceso de aprobación de funcionalidad al enviar actualizaciones realizadas en la aplicación (a menos que declares funcionalidades adicionales).

Si no aprobamos el uso de la capacidad, se producirá un error de su envío para certificación y se harán llegar sus comentarios en el informe de certificación. A continuación, tienes la opción de crear un nuevo envío y cargar los paquetes que no declaren la funcionalidad o, si procede, solucionar los problemas relacionados con el uso de la funcionalidad y solicitar la autorización en un nuevo envío.

> [!NOTE]
> Si su envío usa un espacio aislado de desarrollo en el centro de partners (por ejemplo, este es el caso de cualquier juego que se integra con Xbox Live), debe solicitar aprobación con antelación en lugar de proporcionar información sobre la **opciones de envío** página . Para hacerlo, visita la [página de soporte técnico de desarrolladores de Windows](https://developer.microsoft.com/windows/support). Seleccionar tema de soporte técnico Developer **problema panel**, tipo de problema **envíos de aplicaciones**y Subcategory **otros**. A continuación, se describe cómo se usa la capacidad y por qué es necesario para su producto. Si no proporcionas toda la información necesaria, tu solicitud se denegará. También es posible que se te pida que proporciones más información. Ten en cuenta que este proceso normalmente tarda 5 días laborables o más, por lo que debes enviar solicitud con antelación.
>
> También puede usar este método para solicitar la aprobación (en lugar de proporcionar esta información durante el envío), si no usa un espacio aislado de desarrollo, si lo prefiere confirmar que ha recibido aprobación para utilizar una funcionalidad restringida antes de empezar su Sumisión.

Esta funcionalidad es útil para las aplicaciones que crean o modifican los archivos en su carpeta de instalación.<br /><br />Esta funcionalidad está diseñada para ciertos tipos de juegos de PC de escritorio publicados por Microsoft y nuestros socios. Sin embargo, también podría ser aplicable a otras aplicaciones en algunos casos. |
| **Acciones de instalación personalizada** | El **customInstallActions** capacidad restringida permite que la aplicación declarar el [windows.customInstall](https://docs.microsoft.com/uwp/schemas/appxpackage/uapmanifestschema/element-desktop6-package-extension) extensión en su paquete de manifiesto para que se puede especificar uno o más adicional archivos Installer (.exe o .msi) que se ejecutan con la aplicación. Esto le permite especificar acciones personalizadas para cualquiera de los escenarios de implementación estándar: instalar, actualizar, reparar o desinstalar. Por ejemplo, esto es útil para las aplicaciones que el componente redistribuible parte 3ª de lote.<br /><br />Esta funcionalidad está diseñada para ciertos tipos de juegos de PC de escritorio publicados por Microsoft y nuestros socios. No se concederá para otros escenarios. |
| **Paquetes de servicios** | El **packagedServices** capacidad restringida permite que las aplicaciones creadas por socios de Microsoft y las empresas para declarar el [windows.service](https://docs.microsoft.com/uwp/schemas/appxpackage/uapmanifestschema/element-desktop6-extension) manifiesto de la extensión en su paquete de modo que TI puede instalar uno o varios servicios junto con la aplicación. Estos servicios pueden configurarse para ejecutarse en el servicio Local, servicio de red o las cuentas del sistema Local. Servicios de servicio local y servicio de red solo requieren la **packagedServices** capacidad. Servicios de sistema locales deben tener la **packagedServices** y **localSystemServices** capacidades.<br /><br />No se recomienda que declare esta capacidad en las aplicaciones que se envían a la Microsoft Store. En la mayoría de los casos, no se aprueba el uso de esta funcionalidad.  |
| **Servicios del sistema local** | El **localSystemServices** capacidad restringida permite que las aplicaciones creadas por socios de Microsoft y las empresas para instalar uno o varios servicios de sistema Local, junto con la aplicación (es decir, la aplicación puede declarar el StartAccount para los servicios que LocalSystem). Este escenario también requiere la **packagesServices** capacidad. <br /><br />No se recomienda que declare esta capacidad en las aplicaciones que se envían a la Microsoft Store. En la mayoría de los casos, no se aprueba el uso de esta funcionalidad. |
| **Percepción espacial en segundo plano** | El **backgroundSpatialPerception** capacidad restringida permite que una aplicación tener acceso a la circulación de head del usuario, manos, controladores de movimiento y otros objetos con seguimiento mientras la aplicación se ejecuta en segundo plano. |

## <a name="custom-capabilities"></a>Capacidades personalizadas

El [capacidades restringidas](#restricted-capabilities) sección anterior describe el mismo proceso de aprobación de funcionalidad que puede usar para solicitar aprobación para utilizar una funcionalidad personalizada. El [incrustado SIM](/uwp/api/windows.networking.networkoperators.esim) API son ejemplos de API que requieran una funcionalidad personalizada. Si solo desea ejecutar la aplicación localmente en modo de programador, no es necesario la funcionalidad personalizada. Pero es necesario que publique la aplicación en la Microsoft Store, o para ejecutarlo fuera del modo de desarrollador.

Si tiene un administrador de cuentas técnico (TAM) de Windows, puede trabajar con su TAM para solicitar acceso. Puede encontrar más detalles en [póngase en contacto con su TAM Microsoft](/windows-hardware/drivers/mobilebroadband/testing-your-desktop-cosa-apn-database-submission#contact-your-microsoft-tam).

Para declarar una funcionalidad personalizada, modificar su [manifiesto del paquete de aplicación](https://docs.microsoft.com/uwp/schemas/appxpackage/appx-package-manifest) archivo de código fuente (`Package.appxmanifest`). Agregar el **xmlns:uap4** declaración de espacio de nombres XML y usar el **uap4** prefijo cuando se declara la funcionalidad personalizada. A continuación te mostramos un ejemplo.

```xml
<?xml version="1.0" encoding="utf-8"?>
<Package
    ...
    xmlns:uap4="http://schemas.microsoft.com/appx/manifest/uap/windows10/4">
...
<Capabilities>
    <uap4:CustomCapability Name="CompanyName.customCapabilityName_PublisherID"/>
</Capabilities>
</Package>
```
